# Refresh non-Bevy leaf-crate dependencies

## Summary

Bump `ron` from 0.8 to 0.12 in `Cargo.toml` and run `cargo update` to advance `serde`, `serde_json`, `clap`, and `smallvec` within their existing caret specs. Verify via existing `cargo test` coverage plus one spot-check matrix run. Modifies only `Cargo.toml` and `Cargo.lock`.

## Problem Frame

The project's leaf-crate dependencies have drifted from the versions installed when the project was scaffolded. `serde`/`serde_json`/`clap`/`smallvec` are caret-spec'd and will auto-advance the next time anyone runs `cargo update` for any reason; `ron` is at `0.8` and Cargo treats `0.x` like major versions, so it stays pinned to `>=0.8.0, <0.9.0` until an explicit `Cargo.toml` edit. This is the first dependency PR of the project's life ŌĆö landing it now flattens the noise floor before the higher-stakes work (rand migration, Bevy 0.16 hop) starts moving. (see origin: `docs/brainstorms/2026-05-22-leaf-crate-refresh-requirements.md`)

### U1. Apply dependency edits

**Goal:** Bump `ron` to 0.12 in `Cargo.toml` and refresh `Cargo.lock` so `serde`, `serde_json`, `clap`, and `smallvec` advance to current within their existing caret ranges.

**Requirements:** R1, R2

**Dependencies:** U0 (baseline must exist before edits)

**Files:**
- `Cargo.toml` ŌĆö change `ron = "0.8"` to `ron = "0.12"`
- `Cargo.lock` ŌĆö regenerated by `cargo update`

**Approach:** One-line edit in `Cargo.toml`. Run `cargo update` (without `-p` flags) to pull current patch/minor versions for all caret-spec'd leaf crates. Inspect the resulting `Cargo.lock` diff to confirm only `ron` jumped a major track and the others moved within their `1.x` / `4.x` ranges. **Record the resolved versions** of `ron`, `serde`, `serde_json`, `clap`, and `smallvec` in a scratch note for the PR commit body ŌĆö this is the audit artifact for any future bisect. No source-file changes in this unit.

**Caveat ŌĆö `src/settings.rs` may break compile.** `ron` 0.8 ŌåÆ 0.12 crosses four `0.x` breaking boundaries, and `src/settings.rs:111` calls `ron::ser::PrettyConfig::default()` + `ron::ser::to_string_pretty(...)`. The `PrettyConfig` builder API has shifted across this range. If `cargo build` fails on `settings.rs`, the R6 escalation choice is: (a) update the `PrettyConfig` invocation in `settings.rs` to the current API and treat that single file as in-scope, or (b) pin `ron` at the highest 0.x that still compiles cleanly. Choice (a) is preferred; the `settings.rs` edit is mechanical if needed.

**Patterns to follow:** No prior dependency-refresh PR exists in this repo (verified by Phase 1 learnings researcher ŌĆö `docs/solutions/` contains no library-upgrade learnings). Conventional Rust dependency-bump shape: edit `Cargo.toml`, run `cargo update`, commit both files together.

**Test scenarios:**
- After the edits, running `cargo build` succeeds with no source changes ŌĆö confirms `ron` 0.12 is API-compatible with the existing `ron::from_str` call sites and `RON` struct deserialization patterns.
- `Cargo.lock` diff inspection: `ron` line shows version `0.12.x`; `serde`/`serde_json`/`clap`/`smallvec` lines show version advances within their existing major ranges (1.x or 4.x); no unrelated transitive dependency disappears or appears for surprising reasons.

**Verification:** `cargo build` succeeds. `Cargo.toml` and `Cargo.lock` are the only modified files (`git status` shows nothing else).

## Key Technical Decisions

- **One-line `Cargo.toml` diff, not explicit version-pin tightening.** The leaf crates (`serde`, `serde_json`, `clap`, `smallvec`) already have caret specs that auto-advance via `cargo update`. Tightening them to exact patch versions would be a different decision (more rigidity, more future-PR churn) that the requirements doc explicitly excluded. Only `ron` needs an explicit edit because `0.x` specs are pinned to a single minor track. (see origin: Key Decisions ŌĆö "One-line `Cargo.toml` diff is the correct shape")
- **Spot-check matrix, not byte-identity.** The byte-identity oracle (ideation idea #4) is its own decision and out of scope here. Summary winrate comparison is the verification mode this plan uses. (see origin: Key Decisions ŌĆö "Spot-check matrix run, not byte-identity")
- **No source-file changes.** If `cargo test` surfaces a compatibility break, the PR scope changes ŌĆö escalate rather than fixing inline. Keeping the blast radius narrow is the headline value of this PR. (see origin: R6)
- **Do not bundle with `rand` migration.** The `rand` 0.8 ŌåÆ 0.9 migration (ideation idea #2) is a separate PR. Bundling would blur the blast radius and mix predictive-maintenance dependency work into a hygiene PR. (see origin: Key Decisions ŌĆö "Do not bundle with `rand` migration")

## System-Wide Impact

- **Combat determinism:** No expected impact. `ron` 0.12 parses the same static config files; `serde`/`serde_json` patch advances don't change serialization output for the existing derive macros (verified by the test suite); `clap`/`smallvec` advances are internal-only.
- **Match log format:** No expected impact. The `#[serde(untagged)]` on `RejectionReason` in `src/states/play_match/decision_trace/events.rs:84` is the most version-sensitive surface across the upgrade. `serde` 1.0.220's untagged-enum behavior is the regression vector flagged in external research, but the test suite (`tests/decision_trace_audit.rs`, `tests/decision_trace_builder.rs`) exercises this path on every run ŌĆö a regression would surface immediately.
- **Settings persistence:** `src/settings.rs:111` uses `ron::ser::PrettyConfig::default()` + `ron::ser::to_string_pretty(...)` to write `settings.ron` from the graphical client. The `PrettyConfig` builder API has shifted across the 0.8ŌåÆ0.12 span. If it breaks at compile time, U1's caveat covers the in-scope fix. If it compiles but changes output format subtly, the only visible effect would be future `settings.ron` files written with different whitespace/comment shape ŌĆö gameplay-irrelevant. Detection gap: `cargo test` doesn't exercise this path; it surfaces only on graphical-client save/load.
- **AI authoring loop:** Minor positive side effect. `ron` 0.11+ ships sharper parse-error spans, which help when humans or LLM agents edit `assets/config/abilities.ron` / `items.ron` / `loadouts.ron` and produce a syntax error. Not the headline value but a real downstream benefit.

## Risks

- **`ron::ser::PrettyConfig` API drift in `settings.rs:111` (medium probability).** `ron` 0.8 ŌåÆ 0.12 crosses four breaking boundaries and the `PrettyConfig` builder has been the most-changed surface across this range. Detection: `cargo build` in U1 will catch it. Mitigation: U1's caveat ŌĆö update the invocation to the current API (preferred) or pin `ron` at the highest 0.x that still compiles. If we have to pin, U3 baseline diff still applies and the broader hygiene goal partially landed (the leaf crates still advanced).
- **`serde` 1.0.220 untagged-enum regression on `RejectionReason` (low probability).** The decision-trace audit tests (`tests/decision_trace_audit.rs`) exercise the untagged-enum serialization path on every run. If the patch advance picks up the 1.0.220 regression, the audit will fail ŌĆö that's the expected detection surface. Mitigation: U2's R6 escalation. Rollback: revert the commit.
- **`ron` 0.12 silently changes RON parsing behavior on edge-case syntax (low probability).** Format-stability research and the absence of byte-string literals or exhaustive `Number` matches make this unlikely. Test suite is the detection surface.
- **Matrix winrate shift from non-determinism leak via patch-version advance (very low probability).** RNG is via `rand` (out of scope), and the leaf crates touched here don't sit on the determinism path. If U3's diff against U0 surfaces a winrate shift outside RNG noise, treat as a real regression and investigate before merging.
